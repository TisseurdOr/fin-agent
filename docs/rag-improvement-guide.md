## 1. Query Rewriting（已实现 ✅）

### 原理

检索前让 LLM 把用户口语改写为 2-3 个专业检索词，多路检索后合并去重。

```
用户: "现在炒股应该买什么板块比较好？"
  ↓ LLM 改写
  → "2026年A股配置方向与策略展望"
  → "2026年A股投资主线及机构策略研判"
  → "A股市场板块轮动与行业配置建议"
  ↓ 3 路检索 + 合并去重
  → 12-20 条候选段落
  ↓ Reranker 精排 top-4
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 直面"用户表达 ≠ 文档措辞"这个根本问题 | 每次多一次 LLM 调用，+2s 延迟 |
| 实现简单（约 50 行），无外部依赖 | LLM 改写质量不稳定，偶尔改写跑偏 |
| 对术语鸿沟问题效果显著（如"大模型"→"AI"） | 改写方向可能与文档内容不相关 |
| 可叠加任何其他方案 | 极短查询（<5 字）改写空间有限 |

### 实测数据

| 模式 | 召回率 | 平均耗时 |
|------|--------|---------|
| 纯向量（原始） | 88.9% (16/18) | 0.3s |
| 纯向量（Rewrite） | **100%** (18/18) | 2.3s |
| 生产模式（原始） | 88.9% (16/18) | 2.3s |
| 生产模式（Rewrite） | **100%** (18/18) | 4.5s |

修复的失败案例：
- "大模型概念股有哪些？" → 术语鸿沟（文档用"AI"而非"大模型"）
- "经济下行期哪些行业更抗跌？" → 术语鸿沟（文档用"防御性板块"而非"抗跌"）

### 适用场景

- ✅ 用户群体多样，提问风格差异大（口语化、缩写、行业黑话）
- ✅ 对延迟不敏感的后台/异步/报告生成场景
- ✅ 知识库覆盖面广，同一概念有多种表述
- ❌ 实时对话（每次 +2s 用户等不了）
- ❌ 问题本身已是精准的专业术语

### 关键代码

- `app/rag.py`: `REWRITE_PROMPT`、`_rewrite_query()`、`query()` 的 `use_rewrite` 参数
- `app/main.py`: `AskRequest.use_rewrite` 字段
- `test_recall.py`: 召回率对比测试脚本（18 题，4 种模式交叉对比）

## 2. HyDE（Hypothetical Document Embeddings）

### 原理

让 LLM 先"编"一份假设的答案，用这个答案做向量检索，而不是用原始问题。

```
用户: "经济下行期哪些行业更抗跌？"
  ↓ LLM 生成假设答案
"经济下行期，防御性板块如公用事业、必需消费品、医药等通常表现
 更抗跌。A股策略中建议关注高股息红利资产、低估值蓝筹..."
  ↓ 用这段答案去向量检索
→ 命中真正包含这些概念的文档段落
```

假设答案的向量**天然比问题向量更接近文档向量**，因为：
- 问题和文档之间是"跨域映射"（不同风格、不同词汇）
- 假答案和文档之间是"同域匹配"（都是段落、相似词汇）

### 优缺点

| 优点 | 缺点 |
|------|------|
| 对极短问题效果极好（一个问题 → 一段答案） | 两次 LLM 调用（生成答案 + 最终回答），延迟最大 |
| 和 Query Rewriting 互补而非重叠 | 假设答案可能幻觉，把检索引向错误方向 |
| 假答案包含丰富语义，容易命中 | 如果知识库中没有相关信息，假答案就是纯噪声 |

### 与 Query Rewriting 的区别

| 维度 | Query Rewriting | HyDE |
|------|----------------|------|
| 输出 | 改写为更好的**检索查询** | 扩展为**答案草稿** |
| 长度 | 短（关键词密集） | 长（语义密集） |
| 作用 | 解决"怎么说"（措辞差异） | 解决"找不到"（语义太远） |
| 延迟 | +2s | +3-5s |
| 最佳搭档 | 口语化问题 | 极短/极模糊问题 |

两者是**互补**关系而非替代关系。组合使用：
```
用户问题 → Query Rewrite（3 个检索词）→ [各词分别做 HyDE] → 多路检索
```

### 实现思路（伪代码）

```python
HYDE_PROMPT = """你是一个金融分析助手。请用一段话回答以下问题。
即使不确定也请基于你的金融知识尝试回答。回答约200-300字。

问题：{question}
回答："""

def hyde_retrieve(question: str, vs, k: int):
    # 1. 生成假设答案
    hypothetical = llm.invoke(HYDE_PROMPT.format(question=question))
    # 2. 用假设答案做向量检索
    return vs.similarity_search(hypothetical.content, k=k)
```

### 适用场景

- ✅ 问题极短或极模糊（"下半年怎么看？"、"fintech前景"）
- ✅ 问题缺少文档会用的词汇
- ✅ 作为 Query Rewriting 之后的二级增强
- ❌ 对幻觉敏感的金融合规场景（假答案可能误导检索）
- ❌ 问题本身已经很详细（问题 ≈ 答案草稿）

## 3. 混合检索（Dense + BM25）

### 原理

向量检索做语义匹配，BM25 关键词检索做精确匹配，结果用 RRF（Reciprocal Rank Fusion）融合。

```
            用户问题
           /        \
    向量检索         BM25 关键词检索
    (语义泛化)        (精确命中)
    "大模型≈AI"       "GDP增长5.2%"
           \        /
           RRF 融合
              ↓
          综合排序
```

RRF 融合公式：

```
RRF_score(doc) = Σ 1/(k + rank_i(doc))
其中 k = 60（经验值），rank_i 是 doc 在第 i 个排序列表中的排名
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 无额外 LLM 调用，延迟几乎不增加 | 需要维护 BM25 索引（文档变更时重建） |
| 弥补向量检索在精确数字/代码/专有名词上的弱点 | 中文分词对 BM25 是天然挑战 |
| 成熟方案，工业界广泛验证 | 权重调参需要实验 |

### 适用场景

- ✅ 知识库包含大量精确数字、代码、表格、公式（如 CFA 公式 PDF）
- ✅ 用户常问精确查询（"ROE 公式是什么？"、"2026年GDP增速预测？"）
- ✅ 对延迟敏感、不想加 LLM 调用
- ❌ 纯叙述性文档（如行业报告），BM25 贡献有限
- ❌ 中文文档（BM25 对中文分词不如英文成熟）

### 你的场景

知识库混合了 CFA 公式（结构化）和研报（叙述性），BM25 对前者帮助大。但 542 chunks 规模很小，向量检索覆盖面已经够广，**优先级可以放低**。

## 4. Embedding 模型升级

### 原理

把阿里的 `text-embedding-v2` 换成更强的模型。

| 候选模型 | 提供方 | 特点 |
|---------|--------|------|
| `text-embedding-v4` | 阿里 DashScope | 同体系升级，改动最小，v4 对长文本表现更好 |
| `bge-m3` | BAAI | 多语言，支持 sparse+dense 混合输出，开源可本地部署 |
| `gte-Qwen2-7B-instruct` | 阿里 | 指令微调，可按任务调整 embedding 方向 |

### 优缺点

| 优点 | 缺点 |
|------|------|
| 一行代码改动，零延迟增加 | 提升有限（v2→v4 可能只有 2-5 个百分点） |
| 不增加任何运行时开销 | 需要重建整个向量库 |
| 对一切查询普遍提升，不偏科 | API 调用价格可能更高 |

### 适用场景

- ✅ 作为**基础优化**，在动手做复杂方案前先做
- ✅ 当前模型明显落后（v2 已较老）
- ❌ 想靠升级 embedding 解决所有问题（不现实，提升有天花板）

### 你的场景

当前 chunk_size=800（中文约 400 字），算中等偏长。v2 对长文本的表示能力弱于 v4。**建议顺手升级，一行代码。**

```python
# 改动：只需改一行
embeddings = DashScopeEmbeddings(
    api_key=os.getenv("DASHSCOPE_API_KEY", "sk-placeholder"),
    model="text-embedding-v4",  # v2 → v4
)
```

## 5. Parent Document Retriever（PDR）

### 原理

检索用小 chunk（精准匹配），返回用大 chunk（完整上下文）。

```
建索引时：
  每个 PDF 页面 → 切成 300 字子块（用于检索 embedding）
               → 同时保留 2000 字父块（用于返回给 LLM）

检索阶段：
  用户问题 → 命中小块 chunk_42
           → 找到 chunk_42 的父文档
           → 返回完整 2000 字段落给 LLM
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 返回的上下文更完整，答案质量提升明显 | 改动大：要重建索引、改分块策略、改 query 逻辑 |
| 解决 chunk 截断导致信息丢失的问题 | 返回上下文变大，LLM token 消耗增加 |
| 保留小 chunk 的检索精度 | LangChain 的 PDR 实现有一定学习成本 |

### 适用场景

- ✅ 当前 chunk 太小，经常出现答案被截断在 chunk 边界
- ✅ 需要引用原文的场景（法律、合规、学术）
- ❌ 你的当前场景：chunk=800 + 中文标点分隔符，基本能在语义边界切分，不是瓶颈

## 6. Self-Query Retriever

### 原理

LLM 先提取元数据过滤条件，缩小检索范围后再做语义检索。

```
用户: "2026年A股中期策略里怎么看科技板块？"
  ↓ LLM 提取 filter
{ "source": "2026A股中期策略.pdf", "topic": "科技板块" }
  ↓ 先用 source 过滤，只在目标 PDF 的 chunks 中检索
  ↓ 再对过滤后的候选做向量检索
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 大知识库场景下检索精度大幅提升 | 依赖 LLM 准确提取 filter，提取错了直接搜不到 |
| 支持复杂过滤（时间、来源、类别） | 元数据结构要提前设计，改起来很重 |

### 适用场景

- ✅ 知识库 >100 份文档，检索空间稀释严重
- ✅ 元数据丰富（文档有明确的来源、日期、分类、标签）
- ❌ 你的当前场景：7 个 PDF、542 chunks，简单向量检索 + rerank 足以覆盖

## 决策指南

### 按阶段选

```
当前（542 chunks, 7 PDF）
  ✅ 已做: Query Rewriting            ← 解决了术语鸿沟
  👉 顺手: Embedding v2 → v4 升级     ← 零成本基础优化
  📋 随后: HyDE                       ← 极短查询和口语化的补充
  ⏸️  暂缓: 混合检索、PDR、Self-Query ← 数据量小，边际收益低

中期（50+ PDF, 5000+ chunks）
  🔥 优先: Self-Query Retriever       ← 检索空间大了需要范围约束
  🔥 优先: 混合检索 BM25              ← 结构化公式 + 叙述性报告互补
  📋 考虑: Parent Document Retriever  ← 提升答案完整性

长期（100+ PDF, 10000+ chunks）
  🔥 全套: Rewrite → Self-Query → Hybrid → Rerank → LLM
```

### 按场景选

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 实时对话 (<1s 延迟) | Embedding 升级 + 混合检索 | 不增加 LLM 调用 |
| 报告生成 (可等 5-10s) | Rewrite + HyDE + Rerank | 精度优先，延迟不敏感 |
| 精确查询 (数字/公式) | 混合检索 BM25 | 关键词匹配不可替代 |
| 模糊问题 ("怎么看" / "分析一下") | Query Rewriting + HyDE | 需要扩展语义锚点 |
| 术语不统一 | Query Rewriting | 1→3 个版本覆盖同义表述 |

### 按延迟预算选

```
延迟预算 0ms:   Embedding 升级                      ← 纯免费
延迟预算 500ms: 混合检索 BM25                        ← 关键词索引
延迟预算 2s:    Query Rewriting                      ← 1 次 LLM 改写
延迟预算 5s:    Query Rewriting + HyDE                ← 2 次 LLM 调用
延迟预算 10s:   Rewrite + HyDE + Self-Query + Rerank ← 全链路高精度
```
